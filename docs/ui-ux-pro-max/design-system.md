# vMaker UI/UX 设计系统

来源 skill：`ui-ux-pro-max`

## 简介

vMaker 是一个简约、现代的个人项目展示站。它需要用清晰的信息层级、真实的 GitHub 项目证据、快速的浏览体验和稳定的响应式布局来展示多个个人项目。当前技术栈是 React Router、React、Tailwind CSS、shadcn 风格组件、Lucide 图标和 Motion。

## 产品模式

采用“作品集索引”模式，而不是通用 SaaS 营销页。第一屏应该让访问者立刻知道这是 vMaker，以及这里可以浏览一组个人项目，并尽快看到项目展示区域。

推荐页面结构：

1. 顶部导航：紧凑展示身份、导航和一个主要行动入口。
2. 简短介绍区：说明网站定位，并展示 GitHub 仓库数量、主要语言、最近活跃时间等信号。
3. 精选项目区：使用真实截图或高质量预览图。
4. 项目索引区：支持按语言、topic、精选状态或活跃状态筛选。
5. 简短的关于/过程说明：放在项目证据之后，不要抢占第一屏。
6. 联系与页脚：包含 GitHub、邮箱、部署链接或其他入口。

## 视觉风格

- 主风格：现代简约，但要有作品集的精致感。
- 留白要有节奏，但项目浏览区域不能过于稀疏。
- 优先使用清晰边框、克制阴影和强排版，而不是装饰性背景。
- 主要视觉丰富度来自项目缩略图、截图或界面预览。
- 避免大型抽象插画、渐变光斑、过大的图标卡片和填充式功能区块。

## 色彩系统

使用白色或接近白色的中性背景，高对比文本，再配一个克制的强调色。

推荐角色：

- 背景：纯白或接近白色的中性色。
- 表面：略微抬起的中性色，用于项目卡片和面板。
- 主文本：接近黑色，和背景至少达到 7:1 的舒适对比度。
- 次级文本：中性灰，但必须保持可读，不要过浅。
- 强调色：用于链接、激活筛选项、焦点环、徽标和关键 CTA。

规则：

- 不要让整个网站只由同一色相的深浅变化组成。
- 不要为了“高级感”使用低对比灰色正文。
- 边框在亮色和暗色模式下都必须清晰可见。
- 如果后续增加暗色模式，应使用真正的暗中性背景，不要使用带色偏的猎奇背景。

## 字体与排版

当前项目字体：Inter Variable。

规则：

- 默认保留 Inter 作为主字体，除非后续有明确品牌理由引入第二套展示字体。
- 页面只需要一个强展示标题，不要在卡片内部重复使用 hero 级字号。
- 正文行长建议控制在 55 到 70 个字符左右。
- 元信息使用短标签：分类、年份、状态、技术栈、角色。
- 不要使用全大写正文。大写只适合短标签或非常短的标题。

建议字号：

- Hero 标题：桌面端 48-80px，移动端 36-44px。
- 区块标题：桌面端 28-40px，移动端 24-30px。
- 卡片标题：18-22px。
- 正文：15-17px。
- 元信息：12-14px。

## 布局规则

- 主内容使用一致的最大宽度，通常为 1120-1280px。
- 移动端必须是一等体验，项目卡片在 375px 宽度下不能横向滚动。
- 项目集合优先使用 `repeat(auto-fit, minmax(280px, 1fr))` 这样的自适应网格。
- 不要把卡片套在另一张卡片里。
- 只有在不遮挡内容的前提下，才使用 sticky 或 fixed 导航。
- 区块之间要有节奏，但不能让项目证据被埋到很深的位置。

## 项目卡片规则

每张项目卡片至少应该传达：

- 项目名称。
- 一句话说明项目目的或结果。
- 可视化预览。
- 主要语言和语言构成。
- Topics 或项目标签。
- 最近 push 或最近 commit 时间。
- 代码体量，用 GitHub languages 字节总量表达，不称为精确代码行数。
- 主要链接：打开 Demo/Homepage、查看仓库。

## GitHub 数据规则

第一版只读取 GitHub 用户 `xmtlzzz` 的公开仓库，不接 Neon，不做后台登录。

数据来源：

- 仓库列表：GitHub users repos 接口。
- 语言构成：GitHub languages 接口，值为 bytes。
- 最近提交：GitHub commits 接口，建议每个项目展示最近 3-5 条。
- 展示配置：本地 overrides 文件，用于精选、排序、封面、自定义名称、自定义摘要和隐藏项目。

展示规则：

- “语言构成”用百分比展示。
- “代码体量”用 languages bytes 总和展示，例如 KB/MB。
- 不展示“总代码行数”或“每种语言行数”，除非后续加入构建期统计脚本。
- GitHub description 为空时，用本地 overrides 的 summary 兜底。

交互规则：

- 如果只有一个明确目的地，整张卡片可以点击。
- 如果有多个目的地，使用明确的图标按钮或文字链接。
- Hover 状态应该改变边框、背景或阴影；避免会造成布局位移的缩放。
- 所有可点击元素都需要 `cursor-pointer` 和清晰的焦点状态。

## 动效规则

- 动效用于强化层级，而不是装饰每个区块。
- 微交互动效建议 150-300ms。
- 避免装饰性的无限循环动画。
- 入场动画和文字动效必须支持 `prefers-reduced-motion`。
- Hero 标题动效可以使用，但必须快速结束，不能延迟阅读。
- 项目卡片动效不能造成布局跳动。

## 组件规则

- 可复用 UI 基础组件优先使用 shadcn 风格 CLI 安装。
- 界面操作图标使用 Lucide。
- 不要使用 emoji 作为 UI 图标。
- React Bits 或自定义动效组件应放在独立视觉效果层，不要混进基础 UI 组件。
- 组件应该围绕项目数据复用，而不是为某一张卡片硬编码。

推荐目录：

```text
app/components/ui/           # 基础 UI 组件
app/components/layout/       # header、footer、页面外壳
app/components/sections/     # hero、项目网格、关于、联系
app/components/react-bits/   # 视觉效果和文字动效
app/data/                    # 项目元数据
```

## 可访问性

- 正文至少满足 4.5:1 对比度，主要阅读文本建议达到 7:1。
- 所有图片都需要有意义的 alt 文本，说明项目或界面内容。
- 键盘用户必须可以 tab 到链接、筛选项和卡片操作。
- 焦点环必须清晰可见，并使用强调色角色。
- 状态或分类不能只靠颜色表达。
- 至少测试 320、375、414、768、1024、1440px 宽度。

## Vercel 部署约束

- 优先使用 GitHub API 数据加本地 overrides 配置，不引入数据库依赖。
- 如遇 GitHub API rate limit，可以通过 `GITHUB_TOKEN` 环境变量提高请求额度。
- 动画库要控制使用范围，不要为了单个效果引入沉重依赖。
- 项目预览图需要使用响应式尺寸。
- 除非确实需要，避免引入会复杂化服务端渲染的运行时功能。
- 第一屏要快、稳定，避免明显布局偏移。

## 反模式

- 只有空泛 hero 文案、没有真实项目证据的 AI 风格营销页。
- 没有明确品牌理由的大面积紫蓝渐变。
- 大量结构完全相同、缺少视觉区分的卡片。
- 与项目预览争夺注意力的装饰性动画背景。
- 白底上使用过浅的灰色正文。
- 技术标签比项目价值更显眼。
- 作品集页面把项目列表藏在多个介绍区块之后。

## 交付检查清单

- 没有使用 emoji 作为界面图标。
- 所有可点击元素都有 hover 和 focus 状态。
- 移动端没有横向滚动。
- 项目预览图有稳定的宽高比例。
- 动效尊重 reduced-motion 设置。
- 主文本、次级文本和链接文本对比度都经过检查。
- 项目数据可以更新，而不需要重写布局组件。
- 第一屏同时展示身份和项目证据。
