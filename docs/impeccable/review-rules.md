# vMaker Impeccable 审查规则

来源 skill：`impeccable`

## 审查定位

把 vMaker 当成一个“品牌导向的个人作品集站”，同时它也要具备清晰的项目索引能力。网站应该看起来经过设计，但核心任务仍然是让个人项目容易浏览、比较和打开。

## 目标气质

安静的个人工作室索引：精确、现代、留白充分、有信心。页面应当像被策展过，而不是套模板。项目视觉承担个性，外围 UI 保持克制和严谨。

## 视觉质量标准

- 第一屏必须说明 vMaker 是什么，并展示项目证据。
- 每个主要区块只保留一个主视觉重点。
- 只有在能改善强调或浏览节奏时，才使用不对称布局。
- 避免装饰性填充。空白要显得有意图，而不是没做完。
- 卡片尺寸要稳定，对齐要准确，内部节奏要一致。
- 紧凑界面里的文字必须匹配容器大小，不能使用 hero 级字号。

## 品牌规则

- 简约不等于没有颜色。可以有一个有记忆点的强调色，但要克制使用。
- 技术感不等于到处使用等宽字体。
- 现代感不等于玻璃拟态、渐变光斑或深蓝 SaaS 风格。
- 个人气质应该通过项目选择、截图、文案和细节体现。
- 整体不能看起来像通用作品集模板。

## UX 规则

- 访问者应在 5 秒内理解这个网站包含什么，以及下一步该点哪里。
- 项目浏览应支持通过标题、视觉、分类和技术栈快速扫描。
- 筛选是加速器，不是理解页面的前提。
- 外部链接必须说明目的地：在线演示、代码仓库还是项目说明。
- 空状态、缺失截图和归档项目都需要有设计过的兜底状态。

## 布局审查

实现被认为可接受前，需要检查：

- Header 不占用过多垂直空间。
- 主内容使用一致的容器宽度。
- 项目卡片在每一行都对齐干净。
- 移动端没有横向溢出。
- 没有卡片套卡片。
- 区块间距有节奏，但不会把项目网格埋得太深。
- fixed 或 sticky 元素不会遮挡内容。

## 色彩与对比度审查

- 正文必须舒适可读，不只是勉强通过对比度。
- 次级文本不能变成低对比灰。
- 强调色用于交互和方向感，不要到处当装饰色使用。
- 白色界面上的边框必须可见。
- 如果加入暗色模式，需要单独审查，不能机械反转亮色模式。

## 动效审查

- 可以有一个精致的页面加载或 hero 文字动效。
- 避免每个区块都做重复的滚动淡入。
- Hover 效果不能导致布局位移。
- 动效不能延迟项目浏览。
- reduced-motion 用户必须获得稳定、可读的体验。

## 内容审查

项目文案应回答：

- 这是什么项目？
- 它解决了什么问题，或表达了什么想法？
- 我具体构建了什么？
- 访问者可以打开或查看什么？

避免使用没有具体支撑的空泛表述，例如“现代化解决方案”“强大的体验”“精美界面”。

## Skill 协作规则

- 使用 `ui-ux-pro-max` 定义设计系统规则、技术栈约束和检查清单。
- 使用 `impeccable` 审查层级、审美、视觉信心和最终打磨质量。
- 使用 `React Bits` 做少量精选视觉效果；根据具体视觉任务选择最合适的组件，可以复用本地组件，也可以从 registry 安装更适合的组件。
- 使用 `karpathy-guidelines` 保持范围简单、修改克制，并让成功标准可验证。
- 基础交互组件优先使用 shadcn/base-ui，再考虑自定义交互代码。

## 最终审查清单

- 网站看起来像策展过的项目索引，而不是营销模板。
- 真实项目证据出现在第一屏或接近第一屏的位置。
- 桌面端和移动端都有清晰的排版层级。
- 项目卡片在 hover 前就有用，不依赖隐藏交互才能理解。
- 所有交互元素都有 hover、focus 和键盘行为。
- 视觉效果克制且有目的。
- 页面可以部署到 Vercel，不依赖沉重的运行时假设。
