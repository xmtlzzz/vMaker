# vMaker Karpathy 准则

来源 skill：`karpathy-guidelines`

## 角色定位

这份文档用于让 vMaker 保持简单、清晰、易扩展。它把已安装的 `karpathy-guidelines` skill 应用到 UI 规划、文档和实现过程中。

## 执行规则

- 编码前先思考：明确假设，需求有歧义时先问清楚。
- 简单优先：选择能完成作品集目标的最小设计和最小代码。
- 外科手术式修改：只改和当前任务直接相关的文件。
- 目标驱动执行：先定义成功标准，并在声称完成前验证。

## 原则

- 从最简单有用的页面开始：身份、项目证据、项目链接。
- 在增加复杂交互前，先让项目数据模型清楚。
- 优先写可读组件，不要过早抽象。
- 静态页面已经足够强之后，再添加动画。
- 每个区块都应该回答访问者的一个问题。
- 每个组件都应该只有一个职责，并有清楚的数据输入。

## 产品问题

新增一个区块前，先问：

- 它是否帮助访问者评估项目？
- 它是否帮助访问者浏览项目集合？
- 它是否让后续新增项目更容易？
- 如果删掉它，页面是否真的会变差？

如果答案不够强，就删除或简化这个区块。

## 实现倾向

- 第一版使用 GitHub API 作为项目事实数据源，不引入 Neon、后台或登录。
- 项目展示配置使用本地结构化 overrides 文件，不复制 GitHub 已有事实数据。
- 先做一个高质量的 `ProjectCard`，再考虑多个卡片变体。
- 项目数量不大时，筛选逻辑保持在客户端即可。
- 优先使用适合 Vercel 的静态友好渲染。
- 在本地 React 状态足够前，不引入额外状态管理。
- 不把 GitHub languages bytes 伪装成精确代码行数。

## 反模式

- 在没有真实重复场景前就抽象布局。
- 内容还没稳定就加入复杂动画系统。
- 技术标签比项目结果更醒目。
- 把核心链接藏在只有 hover 才能看到的交互里。
- 为了新奇牺牲清晰的项目评估体验。
